# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
pip install -e .[dev]        # editable install with pytest + Pillow (for the Image widget's tests)
python -m pytest -q          # run the whole suite
python -m pytest tests/test_render.py -q          # one file
python -m pytest tests/test_render.py::test_name  # one test
python -m cozy_tui           # run the built-in demo app (no subcommand → demo)
python -m cozy_tui.themes    # browse/preview every built-in Theme live
python -m cozy_tui.spinners  # browse every built-in Spinner preset live
cozy-tui doctor              # CLI: environment checks; also `demo`, `info`, `--version`
cozy-tui run script.py       # run a script, like `python script.py`
cozy-tui run --debug script.py   # ...with App(debug=True) on, no code change needed
python examples/<name>/<name>.py   # run an example
```

The CLI lives in `cozy_tui/cli.py` (argparse; the `cozy-tui` console script and
`python -m cozy_tui` both call `cli.main`). The demo moved out of `__main__.py`
into `cozy_tui/demo.py`; `__main__.py` is now a thin dispatcher. `__version__`
is exposed from `cozy_tui/__init__.py` (via `importlib.metadata`, `"dev"` when
running from an uninstalled source tree). `cozy-tui run` (`cli._cmd_run`) executes
the target file via `runpy.run_path(..., run_name="__main__")` with its own
directory prepended to `sys.path` and the remaining CLI args forwarded as
`sys.argv[1:]`, matching `python script.py`; `--debug` just sets `COZY_TUI_DEBUG=1`
before running it.

There is no linter/formatter config, but the code is `black`/`isort`-formatted (note the `# fmt: off` blocks in `events.py`).

User-facing docs live in `docs/` (`concepts.md`, `widgets.md`, `layouts.md`, `styling.md`, `interaction.md`, `examples.md`) and are linked from `README.md` — when a change alters public behavior, the relevant `docs/` page is the place it gets described. `_internal/` is git-ignored scratch space, not part of the package.

### Test harness (`testing.py`)
`cozy_tui.testing.Harness(app)` is the **public** way to drive an app headlessly — what library *users* write tests against, and the thing to reach for in new tests here rather than hand-rolling `_dispatch_input` calls. It forces `full=False` (a full-screen app would size its buffer from the real terminal, making assertions machine-dependent) and `catch_errors=False` (the default would swallow a test failure into a crash view that then hangs on a terminal that isn't there). Every action re-composes via `App._compose()`, **not** `render()` — the buffer gets filled without emitting to the terminal, so a test run isn't interleaved with escape sequences; the trade-off is that the diff renderer isn't exercised, which is deliberate (this harness tests apps, not the renderer). Time is virtual: `advance(seconds)` accumulates a private clock offset and calls `App._drain_timers(monotonic() + offset)`, so a 30-second timeout costs nothing. Workers are real threads, so `settle()` polls `_drain_workers()` up to a timeout and delivers callbacks on the calling thread, matching the loop's guarantee. Actions return `self` so they chain.

### Testing conventions
Tests are pure-Python and headless — **never** require a real terminal:
- **Default to the `Harness`** (above). The house pattern is a `make_ui(**kw)` helper returning `Harness(App(full=False, size=…, …))`; tests that also assert on library internals take an `app = ui.app` alias on the next line and use the harness for interaction (`ui.press`/`ui.click`/`ui.hover`/`ui.compose`) and reading (`ui.screen`/`ui.line(n)`/`ui.cell(c, r)`). This is what every interaction-level test file now uses.
- **Don't call `app.render()`** unless the test is specifically about what reaches the terminal — the renderer itself (`tests/test_render.py`), or an assertion on emitted escape bytes (`tests/test_mouse.py`'s `?1003h` motion-upgrade check, `tests/test_traceback_view.py`). `render()` writes escape sequences to stdout, which is why the suite used to interleave thousands of them with pytest's output; the harness composes instead. Seven deliberate `render()` calls remain, and each is that kind of test.
- Reading the screen (`ui.screen`/`lines`/`cell`) **composes first**, matching `App.snapshot()`. Beyond freshness, parts of this library settle over two passes (a `Tabs` panel or docked `Box` starts at a placeholder size and gets its real one from `dock_resize` *during* a draw).
- **Animations need `ui.advance(...)`**, not just timers: `Tabs` deliberately draws no content while its underline glides, so a test that switches tabs and reads immediately sees an empty panel — correctly, since a user would too. `advance()` moves a virtual clock that `_CLOCK_MODULES` (motion/tabs/button/spinner/animated_label) read through, so nothing sleeps. Several DevTools tests only passed before because they switched tabs *before* the first draw, never starting the glide.
- Pure-logic tests (widths, style, event parsing, easing, game rules) need no app at all — no harness there either.
- Example apps that live outside the package (`examples/aquarium/aquarium.py`, `examples/game_2048/…`) are loaded via `importlib.util.spec_from_file_location` and exercised as **pure logic** — no `App`/`Widget` involved (`tests/test_aquarium.py`, `tests/test_game_2048.py`).
- Clipboard tests self-skip when no platform backend is available (see `tests/test_clipboard.py`); CI runs them under Xvfb/sway to exercise the X11/Wayland backends for real.

## Architecture

`cozy_tui` is a from-scratch TUI library. The only *required* third-party dependency is `rich` (used to render `Markdown`/`MarkdownInput` and to syntax-highlight `TracebackView`/`crash_screen.show_traceback`); everything else — input parsing, clipboard, unicode width — is hand-rolled on the standard library. `Pillow` is an *optional* dependency (`pip install cozy-tui[image]`), needed only by the `Image` widget — see "Image widget" below for how it stays lazily imported so the base install never requires it. Targets Python 3.10+.

### Render loop (`app.py`)
`App.run()` is the single event loop. Each frame:
1. `render()` clears an in-memory cell `buffer` (a `rows × cols` grid of `Cell`), applies docks, then calls `widget.draw(self)` on every widget, then overlays.
2. Widgets paint by calling `app.write(x, y, text, style)` — the canvas is the `App` itself.
3. A **double buffer** (`_prev_cells`) enables diff rendering: `_do_diff_render()` emits VT escapes only for changed cells; `_do_full_render()` repaints everything (forced via `invalidate()` after screen switches / resize). Call `invalidate()` whenever you change something the diff can't detect.

Between frames the loop blocks in `wait_input()` (never busy-spins) and wakes for input, the cursor blink (`BLINK_INTERVAL`), terminal resize, `tick_interval`/animation frames, or worker results.

`App(catch_errors=True)` (the default) wraps the loop in `except Exception`: the terminal is restored exactly as on a clean exit (same `finally`), then `crash_screen.show_traceback(exc)` runs as a *separate* full-screen crash view — deferred until after the `finally` so it never starts before this app's raw mode/alt screen/mouse tracking is fully torn down. `KeyboardInterrupt`/`EOFError` are handled first and never reach the crash screen. Pass `catch_errors=False` for a script/test that wants `run()` to raise normally instead (or that has no real terminal for the crash screen to block on). `show_traceback`'s own internal `App` always sets `catch_errors=False` — otherwise a crash *in* the crash screen would recurse into showing a crash screen for itself, without end.

### Coordinate system
- Widget `x`/`y` are in **terminal cells**, relative to the parent (`abs_x`/`abs_y` walk the parent chain).
- Widget **sizes** passed as `"WIDTHxHEIGHT"` strings are in *virtual pixels*; divide by `App.SCALE` (10) to get cells. `Box.natural_width` = `width // scale + 2` (border eats 2 cells).
- `write()` accounts for `scroll_y` (vertical scroll of the base UI) unless an overlay is drawing (overlays are screen-fixed).

### Widgets (`cozy_tui/widgets/`, grouped by kind)
All widgets subclass `Widget` (`widget.py`) and implement `draw(canvas)`; interactive ones set `focusable = True`, handle `on_key(key)`, and expose a `cursor`/`cursor_style` for the real terminal cursor. Layout containers (`VBox`/`HBox`/`Grid`) subclass `Layout` (`widgets/layout/layout.py`) and implement `_arrange()` to auto-position children. Package dirs: `display/`, `input/`, `layout/`, `selection/`. Public API is re-exported from `cozy_tui/__init__.py`.

`Layout.add(widget, flex=N)` marks a child to share leftover space along the main axis once the layout is docked to a known size: `dock_resize(w, h, scale)` records that target size (`_target_w`/`_target_h`), and `Layout._flex_extras()` — shared by `VBox`/`HBox`'s own `_arrange()` — distributes whatever's left after every child's natural size and inter-child gaps are subtracted, proportional to each flex-marked child's weight against the sum of all flex weights. `flex=0` (default) is unaffected: fixed natural size, same as before flex existed. Undocked layouts (`_target_w`/`_target_h` still `None`) ignore `flex` entirely and keep the old shrink-to-fit behavior.

### Screens (`screen.py`)
`app.screen(name)` get-or-creates a `Screen` — a named top-level widget list — and `app.show(name)` swaps which one the app draws. Deliberately **not** a class to subclass and not a builder function: a `Screen` owns a plain `widgets` list plus a remembered `focused` widget, so switching away and back preserves a half-typed form, a scroll position, and the focus, where a `build()` re-run on every switch would silently discard all of it. `screen.add`/`dock`/`remove`/`focus` delegate to the App's own equivalents so there's no second API to learn.

The swap itself is just `self.widgets = screen.widgets` (same list object, so `app.add` keeps working and identity is preserved across switches) plus `invalidate()` — a new screen shares nothing with what the diff renderer last emitted. Focus is cleared *before* the swap so the outgoing screen's widget gets its `on_blur`, then restored from `screen.focused` (falling back to the first stop when that widget is gone). `_content_list()`/`_install_content()` exist because `toggle_devtools` moves the real widget list into `_AppContentPane.children`; routing every read/write of "the app's top-level widgets" through that pair is what makes a screen switch work identically whether or not DevTools is open. The **first** screen created adopts whatever is already in `self.widgets` and starts out current, so screens can be added to an existing app without blanking it or reordering setup code. An app that never calls `screen()` is completely unaffected (`current_screen` stays `None`).

### Widget lifecycle: `remove`, `visible`, `disabled`, focus events
The four post-construction operations, deliberately built as one change because they share the same three choke points (`_focusables_in`, `_hit_widget`, `_set_focused`).

`Widget.visible`/`Widget.disabled` are **plain attributes**, not properties — a property would have to reach back into an `App` the widget doesn't know about. Enforcement therefore lives at the choke points instead: `_focusables_in` returns `[]` for an invisible subtree and skips a disabled widget itself (a disabled *container* stays inert while Tab still reaches its children); `_hit_widget` returns `None` for an invisible subtree; each container's child-draw loop skips invisible children; and `VBox`/`HBox._arrange()` measure a hidden child as `(0, 0)` *and* don't advance the cursor, so its surrounding `gap` collapses too (`Grid` deliberately keeps the slot — reflowing around a hidden cell would move every other cell). The one case attributes can't cover is a widget hidden/disabled **while focused**, caught in `_dispatch_input` before any key is routed.

A disabled widget under the cursor makes `_hit_widget` return the module-level `_BLOCKED` sentinel rather than `None`: `_hit_test` translates that to "no target, stop searching", so the click dies on the disabled widget instead of falling through to whatever is behind it. `None` would have meant "keep looking", which is right for a *hidden* widget and wrong for a disabled one.

Dimming is free for every widget because **`Widget.style` is a property** that returns a dimmed copy while `disabled` — every widget already reads `self.style` in `draw()`, so none of them had to learn the flag exists. The copy is cached against the base style's `(fg, bg, styles)` triple rather than its identity, for the same reason the markup widgets are: a theme switch re-colors that object in place (`App._sync_theme_style`), and an identity check would strand a disabled widget on the old theme.

`App.remove(widget)` finds the owning container anywhere in the tree and, crucially, **moves focus off whatever it takes away** — a removed widget that keeps `app.focused` still receives every keystroke while being invisible and un-Tab-able, which presents as a UI that silently stops responding. The successor is resolved *before* detaching so "the next stop" means what Tab would have chosen. `Widget.remove`/`Widget.clear` are the bare container versions (any widget with `.children`), with `Layout._children_changed()` re-dirtying the cached arrangement.

`Widget.on_focus`/`on_blur` fire from `App._set_focused` — already the single assignment point — *after* `self.focused` is updated, so a handler inspecting it sees the final state. Re-focusing the same widget is a no-op.

### Focus & hit-testing
Tab order comes from `_collect_focusables()` → `_focusables_in()`, which **dives into focusable descendants**: a focusable container is only itself a Tab stop when it has no focusable children. Mouse `_hit_test()` walks widgets top-to-bottom, children-first, and clicking a container focuses its first focusable child (matching Tab). A modal overlay confines both Tab and mouse to itself.

Mouse events (`events.py`: `MouseClick`/`MouseDrag`/`MouseRelease`/`MouseMove`) are routed by `App._dispatch_mouse()`: click/move hit-test under the cursor, drag/release go to `self.focused` (press-to-drag capture). Widgets get `on_mouse_click`/`on_mouse_double_click`/`on_mouse_drag`/`on_mouse_release`/`on_mouse_move`/`on_mouse_enter`/`on_mouse_leave` (base no-ops firing the `on_click`/`on_drag`/`on_enter`/… registration callbacks). Double-click is synthesized in `_dispatch_click` (`DOUBLE_CLICK_INTERVAL`). Hover (`MouseMove`) is opt-in **per widget** (`widget.mouse_moves = True`, or registering `on_hover`/`on_enter`/`on_leave` — there is no app-wide switch) — the app enables VT any-motion tracking (`?1003h` vs `?1002h`) automatically once at least one live widget wants it, and tracks `_hovered` to dispatch enter/leave on boundary crossings. `app.on_mouse(handler)` is a global pre-dispatch hook that can consume events.

Keyboard/mouse routing for a single already-parsed event lives in `App._dispatch_input()` (modal capture, Tab cycling, Ctrl+C, scroll keys, registered handlers, then the focused widget) — `run()`'s loop just feeds it parsed events, so any alternate frontend with its own event source can reuse the exact same routing instead of duplicating it. One quirk worth knowing: inside its modal branch, **F12 always reaches `toggle_devtools()`** rather than being treated as a second "close this modal" key alongside Esc — otherwise F12 could never open Cozy DevTools while a modal is open.

### Overlays / z-layer
`app.open_overlay(widget)` pushes onto `_overlays` (drawn above the base UI). **Modal** overlays capture all keyboard/mouse input and (with `dim`) grey the background via `_apply_backdrop()`; non-modal ones are purely visual by default (`Tooltip`, a plain `Toast`). `app.prompt(...)`/`app.confirm(...)`/`app.pick_file(...)` are modal dialogs built on this (`PromptDialog`/`ConfirmDialog`/`FilePicker`); `ThemePalette`/`CommandPalette` (bound to Ctrl+T/Ctrl+P by default, see "Theming" below) and `RightClickMenu`/`MenuBar`'s dropdowns are the same pattern. Every one of these is a **single self-contained widget** — its own text buffer, its own Up/Down/Enter navigation, its own `draw()` — rather than composed from a real `Input` + `ListView`: a modal routes *every* key to one focused widget with no bubbling, so splitting the text-entry half and the list-navigation half across two child widgets would need a hand-written wrapper deciding which key goes where anyway, for no less code than handling it directly. Overlays position in screen space (no scroll), re-centered each frame if `center=True`.

Non-modal overlays are otherwise **mouse-transparent**: `_mouse_target()`/`_hit_test()` only ever look at `self._overlays` when a *modal* is open, so a click/hover normally passes straight through a non-modal overlay to whatever's underneath. `App._hit_non_modal_overlay()` is the one exception — called from `_dispatch_mouse` only when no modal is open, it gives non-modal overlays (topmost z-layer first) a look at a left click/hover *before* falling through to the base UI, routing straight to the target's `on_mouse_click`/hover handlers **without** touching `app.focused` (reusing the normal `_dispatch_click` path would silently steal focus via its `_set_focused()` call — the one thing a non-modal overlay must never do). `Toast(actions=[(label, callback), ...])` (usually via `App.toast(..., actions=...)`) is the one consumer today: an actionable toast renders a `[ Label ]` button row, positions itself fresh every frame via `_place()` rather than `self.x`/`self.y` (so it caches `self._bounds` from its own last `draw()` for `contains()`/hit-testing, the same pattern `Box._bounds` uses), and pauses/restarts its auto-dismiss timer on hover (via the ordinary `on_enter`/`on_leave` opt-in) so reaching for a button doesn't get the toast pulled out from under the cursor. A plain (no-`actions`) toast still swallows clicks landing on its own footprint now too (correct top-z-layer behavior) even though nothing on it is clickable.

`ThemePalette`/`CommandPalette`/`FilePicker` all share their query-filtered, scroll-clamped list logic via `_search_palette._SearchPaletteMixin` (`self._all` → `self._matches` via `_refilter()`; Up/Down/Home/End/Enter/typing all live in the mixin's `on_key`) — a subclass need only supply `_matches_query(item, query)` and `_activate_item(item)`. `SearchBar` (`widgets/selection/search_bar.py`) is the same mixin's non-modal sibling: an embeddable, borderless (unlike the three modals) widget you `.add()` into your own layout instead of opening as an overlay. `fuzzy_searching=True` switches the mixin from plain substring filtering to `fuzzy_score`-ranked subsequence matching (also in `_search_palette.py`) — opt-in per widget via `self._fuzzy`, so the three existing modals are untouched unless they set it too.

### Dock layout (`_dock.py`)
`app.dock(widget, "top"/"bottom"/"left"/"right"/"fill")` (and `Box.dock(...)` for docking inside a box) reserves an edge-anchored band sized to the widget's natural size, shrinking the remaining fill region for whatever docks next; each docked widget with a `Layout` gets `dock_resize(w, h, scale)` called with its final slice so flex children can react. Order of `dock()` calls matters — each new dock carves from what's left after the previous ones.

### Theming (`theme.py`, `_presets.py`)
A `Theme` bundles the colors the shared visual language draws from: the base `App.style`, `accent`, `success`/`warning`/`error`/`info` (what `Toast` reads its color from), `muted`, and `selection_fg`/`selection_bg` (what `style.py`'s `selection_style()` — shared by `ListView`, `RadioSet`, `CheckList`, `Table`, `Tree`, `Dropdown`, `Checkbox`, `RightClickMenu`, `Slider`, `MenuBar` — builds its focused-row highlight from). Built from a built-in `mode` (`Theme.MODES`, `_presets.py`; over 30 presets, grouped into categories for the browser/palette) and/or per-role overrides. There is exactly **one process-wide active theme** (module-level global state in `theme.py`): `get_theme()`/`set_theme(theme)`/`theme.activate()`. `App()` built with no explicit `style=` picks up the active theme's `style` (a fresh copy, not the same object — mutating one app's `.style` must never leak into another's). **Switching is live.** The active theme is a `State` (`theme._active_theme`), so `set_theme()` notifies subscribers; `on_theme_change(cb, owner=)`/`unsubscribe_theme(cb)` is the public hook. `App` subscribes when its style came from the theme (`_theme_styled`, i.e. no explicit `style=`) and `_sync_theme_style()` **mutates `self.style` in place** rather than replacing it — that's the trick: every widget built with `style=app.style` holds that exact object and re-colors for free, with no registry of borrowers. Mutation is safe because the Style is already this app's private copy. It then calls `invalidate()`, since the diff renderer compares against what it last *emitted* and unchanged glyphs would otherwise keep the old colors. An explicit `App(style=...)`/`Widget(style=...)` is never overridden — a theme switch must not discard a deliberate choice. Everything resolved at draw time follows automatically (`selection_style()`, the modal scrim, `Toast`, the dialogs' accent); `Box`'s focused-border style and `Bindings`' four styles were construction-time caches and are now draw-time properties for the same reason. Note the bound-method trap this surfaced: `subscribe(self.method, owner=self)` would pin `self` forever, so `State.subscribe` detects that case and stores a `weakref.WeakMethod`. `App.open_theme_palette()` (Ctrl+T) is a searchable `ThemePalette` over `Theme.MODES`; `cozy_tui.themes` is a standalone categorized Tree-based browser (`python -m cozy_tui.themes`) for previewing every preset live. `App.cycle_theme()` is a plain next-mode advance, not bound by default. `App.open_command_palette()` (Ctrl+P) is the same `CommandPalette` pattern generalized to arbitrary actions, fed by `App.register_command(name, callback, description=)` — `App` itself registers "Quit"/"Change Theme"/"Keys" (and "Toggle DevTools" when `debug=True`) this way, so overriding one of those names replaces it.

### Debugging: Cozy DevTools (`App(debug=True)`, `_devtools.py`)
`app.debug(*values, sep=" ")` is a `print()`-equivalent that's safe to call while the raw-mode/alt-screen loop is running (a bare `print()` would corrupt the display) — it appends to a bounded ring buffer (`_debug_log`, `deque(maxlen=500)`) instead. It's a no-op, and the buffer isn't even allocated, unless the app was built with `debug=True`; `debug=None` (the constructor default) resolves from `COZY_TUI_DEBUG=1`, which `cozy-tui run --debug script.py` sets — an explicit `True`/`False` always wins over the env var. `debug_log_path=` additionally tails each line to a file. `default_logs=True` (the default, only meaningful when `debug=True`) has `App` log its own focus changes, key presses, and mouse clicks/drags via `app.debug()` automatically — mouse-move/hover is deliberately excluded (fires far too often to be useful).

**F12** (`app.toggle_devtools()`) is the *only* debug key — one Chrome-style panel instead of three separate ones, and (unlike its first version) not an overlay at all: a real docked pane split from the app's own content by a draggable `Splitter` (drag the bar, or focus it and use Left/Right, exactly like any other `Splitter`), so the app visibly shrinks to share the screen instead of being covered. `toggle_devtools()` wraps whatever's currently in `self.widgets` in `_devtools.py`'s `_AppContentPane` — which re-runs `_dock.py`'s `dock_layout()` scoped to whatever rectangle `Splitter` assigns it, so the app's own top/bottom/fill docks (a MenuBar, a header, a footer, a filled `Tabs`) correctly reflow into a narrower width, the same way a real browser page reflows when DevTools opens — then builds `Splitter(content_pane, devtools_panel, ratio=0.62)` and docks *that* as the sole entry in `self.widgets` (`self.dock(splitter, "fill")`, not a bare append, so a terminal resize is tracked live same as any other fill dock). `App.add()` redirects into `_devtools_content_pane.children` while this is active, so anything the app adds mid-session still lands in the right place; closing restores `self.widgets` from that pane (not the original snapshot, in case something was added meanwhile) and whatever was focused before F12 opened. `DevToolsPanel`'s own tab-switching is a plain `Tabs` instance (`self.tabs`), with five tabs:
- **Elements** — a live property panel (`_ElementsView`) for `App._selected_widget`, following the mouse via `App._hit_any()` (a hit-test that, unlike `_hit_test()`/`_hit_widget()`'s default, doesn't require `focusable`, so a plain `Label` or decorative `Box` is inspectable too, and recurses through anything with `.children` including the Splitter/`_AppContentPane`) and drawing an accent-colored highlight border (`draw_highlight()`, called from `_compose()` between the base-widget draw pass and `_draw_overlays()`) around whatever it's pointing at.
  - **Live editing** (`_devtools_edit.py`) sits under that property list: `build_snippet(widget)` synthesizes an editable constructor-style snippet from the widget's *current attributes* (never its real source — nothing reads the file it was constructed in), a persistent `CodeInput` + Apply/Revert `Button`s edit it (a `CodeInput`, not a plain `Input`, so the snippet reads as highlighted Python while unfocused — `background=False` so it doesn't paint a theme slab inside the panel's chrome), and `apply_snippet(widget, text)` assigns the parsed fields back onto the live object, with the next ordinary frame showing the result. Parsing is **structural, never `eval`** — `ast.parse(mode="eval")` checks the shape (one call, keywords only) and `ast.literal_eval` handles each value, so an edit can only ever supply plain literals; `Style(...)` is the single callable form, matched by name structurally (no namespace lookup) because colors are the thing most worth poking at live. Fields are validated *before* any is assigned (all-or-nothing — a half-applied edit on a live UI is indistinguishable from a working one), unknown names are rejected rather than silently `setattr`'d, and `_kind()` blocks a type-breaking edit (`x='wide'`) while allowing int→float. Which fields appear is driven by one shared table (`_POSITION + _FIELDS`) filtered by `hasattr` + "is a literal", so there's no per-widget-class registry. The editor's three widgets are built **once** in `_ElementsView.__init__` and only repositioned/refilled by `_rebuild()` (which recreates the property `Label`s) — rebuilding them too would hand `App.focused` a discarded orphan whenever a rebuild fired while the editor was focused. Focusing the editor sets `_selection_frozen` (else a stray `MouseMove` retargets Elements and rebuilds the snippet mid-edit), and `_dispatch_input`'s Esc-to-unfreeze branch excludes `self.focused is self._devtools_editor` for the same reason — Revert, not Esc, discards an edit. Errors surface in the panel via `EditError`, never raised out of the Button callback (cf. `_quick_screenshot`'s toast-don't-raise handling).
- **Console** — the live debug log (`_LiveLog`, moved over unchanged from the old standalone F12 pane).
- **Performance** — FPS (`App._current_fps()`, rolling over `_frame_times`), live widget count (`App._count_widgets()`), focused/hovered widget, last mouse position, and frame/render/layout timings; rebuilt every `draw()` (unlike the seq-gated Elements/Console tabs, these numbers are expected to change every frame).
- **Coordinates** (`_CoordinatesView`) — a live readout of `App._last_mouse_pos` (already tracked on every `MouseMove` whenever `debug=True`, regardless of which tab is active). Unlike Elements, a click here is a one-shot action, not a mode: `_dispatch_mouse`'s `coords_active` branch (same guard shape as `elements_active`) copies `x=.., y=..` straight to the clipboard (`cozy_tui.clipboard.copy`) and shows a confirmation `Toast`, swallowing the click so it never activates the real widget underneath. Deliberately doesn't touch `MouseMove` at all, unlike Elements — hover/tooltips on the real app keep working normally while this tab is open, since there's no selection to preview.
- **Tree** (`_build_widget_tree()`) — a read-only, always-expanded snapshot of the app's real content (`app._devtools_content_pane.children`, *not* `app.widgets` — that's just `[the Splitter]` while DevTools is open) under a synthetic `"App"` root, built once per switch-into (not continuously, so there's no expand/collapse state to lose across rebuilds). Clicking a node sets `App._selected_widget`/`_selection_frozen` (so Elements shows it and the highlight border picks it up) without switching tabs away from Tree — deliberately: bouncing to Elements on every click made it hard to click through several nodes in a row. This is the one place selection is shared across tabs today (a natural follow-up, not yet built: Performance showing per-widget redraw/event counts for whatever's selected).

The whole panel is **non-modal**, always — Elements needs to hover/click the *real* app underneath it (now beside it), which a modal overlay could never allow. Because DevTools' own tab bar/Tree are therefore real, focusable widgets reachable through the ordinary `_hit_test()`/`_dispatch_click()` path (not overlay plumbing), clicking them Just Works via the same mechanism every other focusable widget already uses — no special-casing needed: `_dispatch_click` finds them and calls `_set_focused()` itself, so the clicked tab/row gets the normal focused-widget highlight and, as a bonus, keyboard routing (Left/Right then switches tabs via `_TabBar.on_key`). `_dispatch_mouse`'s `elements_active` branch (gated on `modal is None`, `App._devtools_tabs.selected_title == "Elements"`, the event not landing on the panel's own chrome — `_hit_widget(self._devtools_panel, ...)` — *and* not landing on the Splitter's own divider bar, whose `contains()` only ever matches the bar itself, never either pane's area) takes over the mouse entirely while active: a `MouseMove` retargets, a `MouseClick` freezes onto that widget *without* firing its real click handler. Esc only unfreezes back to live tracking (`_dispatch_input`); F12 is the sole way to close the panel — a deliberately asymmetric Esc so it can't be confused with a modal's close key. `_hit_non_modal_overlay()` (the mechanism `Toast`'s action buttons use to be clickable despite non-modal overlays otherwise being mouse-transparent) is unrelated to any of this now that DevTools isn't an overlay — it's Toast/Tooltip-only by construction. `DevToolsPanel.draw()` calls `canvas.request_frame(1/20)` unconditionally while open — the same mechanism `AnimatedLabel` uses to keep the loop ticking — so its status bar (`FPS: … | Widgets: … | Focus: … | Theme: …`, always visible regardless of active tab) and Performance tab stay live even when the app is otherwise idle; closing it lets the app go back to fully event-driven, zero-CPU-at-rest rendering. All render/layout timing (`App._last_frame_ms`/`_last_render_ms`, `Layout.draw()`'s `canvas._debug_layout_ms` accumulator) is computed only when `self._debug_log is not None`, so a non-debug app pays a single attribute check, never a `perf_counter()` call.

`Key.F3` and `Key.CTRL_SHIFT_D` (the latter a real, distinguishable combo — see `events.py`'s CSI-u carve-out, the same modifyOtherKeys/kitty-protocol mechanism `Key.CTRL_Y`/Ctrl+Shift+Z uses) still exist as ordinary parseable keys; neither is bound to anything by default anymore.

### Input parsing (`events.py`) & platform backend (`_console.py`)
**Cursor shape** is DECSCUSR (`CSI Ps SP q`), tabled in `ansi.py` as `CURSOR_SHAPES`/`cursor_shape_esc(style, flash)` with `TERMINAL_CURSOR_STYLES` as the single source of truth for "the terminal draws this one". All three built-in `Input.cursor_style` values (`block`/`underline`/`vertical`) are terminal-drawn, each with a blinking (odd code) and steady (even code) variant selected by the widget's `flash`. Because the terminal owns the blink, `App._cursor_esc()` deliberately does **not** consult `_cursor_on` — toggling `?25h`/`?25l` on top of a natively-blinking cursor would fight it and blink at our rate rather than the user's configured one — so the escape stays constant and the loop's `BLINK_INTERVAL` wake writes nothing for these widgets. `_cursor_on` still drives the fallback path: a widget with some *other* `cursor_style` gets no terminal cursor and paints its own caret into the cell buffer (`Input._cursor_style_obj`, guarded in `_input_draw.py` by the same `TERMINAL_CURSOR_STYLES` set so a terminal-drawn style can never be painted twice).

`events.py` parses raw VT byte sequences into `Key.*` constants, `MouseClick`/`MouseDrag`, and bracketed-`Paste` events — this is the portable core. `_console.py` isolates the *only* platform-specific parts: entering raw+VT mode and blocking for input. Windows uses the Console API (`ctypes`/`kernel32`); POSIX uses `termios`/`tty`/`select`. Keep platform-specific code confined to `_console.py`.

### Unicode width (`_width.py`)
A built-in `wcwidth`-style layer: `char_width(ch)` returns 0 (combining/ZWJ — dropped by `write`), 1, or 2 (CJK/emoji — the trailing cell is blanked to stay grid-aligned). This keeps the cell grid honest without a `wcwidth` dependency.

### Image widget (`widgets/display/image.py`)
Renders a raster image via 2x2 "quadrant" truecolor blocks: each cell packs *four* source pixels, split by luminance into two representative colors (fg/bg), drawn with whichever of the 16 Unicode Block Elements glyphs (`▘▝▖▗▀▄▌▐▚▞▛▜▙▟█` + space — all in the same widely-supported `2580–259F` range, confirmed width-1 in `_width.py`'s tables) best matches which quadrants are the brighter half (`_quadrant_cell()`) — double the effective resolution of plain upper-half-block rendering, in both directions. Colors are plain `Style(fg="rgb(r,g,b)", bg="rgb(r,g,b)")` — `ansi.py`'s existing `_color_codes()`/`_truecolor_sgr()` already downgrade `"rgb(...)"` to 256/16-color at the active depth, the same reuse principle as the SVG/HTML export's `resolve_rgb()` (no separate color table). Sizing follows `Box`/`ScrollView`/`Splitter`'s `"WIDTHxHEIGHT"` virtual-pixel-string convention (`resize("800x700")`, ÷ `App.SCALE` for cells); with no explicit size, it auto-fits to a default `_DEFAULT_COLS`-cell width with height derived from the source's own aspect ratio times `_CELL_ASPECT` (0.5 — terminal cells are ~2x taller than wide, so this keeps a square/wide source from rendering into a visibly stretched box; this factor applies regardless of the sub-cell technique, unlike an earlier version of this widget which mistakenly assumed half-block's 2x vertical sub-sampling canceled it out).

**Caching is the point of this widget** — `draw()` never re-runs Pillow's resize/pixel-sampling every frame; `_rebuild_cache(cols, rows)` (a rows×cols matrix of prebuilt `(glyph, Style)` pairs) only reruns when the target cell size actually changes or a mutator (`load_img`/`crop`/`blur`) sets `self._dirty`. A Pillow-esque fluent builder (`load_img`/`resize`/`crop`/`blur`/`save`/`render`, every mutator returning `self`) sits alongside the plain `Image(x, y, "cat.png")` constructor form. `crop(top=, left=, bottom=, right=)` trims pixels off each named edge (not Pillow's absolute box tuple). `Pillow` is lazily imported (`_ensure_pillow()`) only inside methods that actually touch pixels — never at module import time — so `from cozy_tui.widgets import Image` always succeeds even without Pillow installed; only calling `load_img`/the constructor with a source raises, with an install hint (`pip install cozy-tui[image]`).

### Inline markup (`markup.py`)
`"[bold red]Error[/] connecting"`, opt-in per widget via `markup=True` (`Label`, `Text`, `AnimatedLabel`, `Log` — off everywhere by default, so existing text containing brackets can never change meaning). Hand-rolled, **not** routed through Rich's markup: a tag body is validated word-by-word against `ansi.py`'s own `_FG`/`_ST` tables plus `#rrggbb`/`rgb(...)`/`color(N)`, so markup speaks exactly `Style`'s color vocabulary (truecolor included — going through `_rich_bridge` would have collapsed every color to one of 16 ANSI names first). `[/]` and `[/name]` both pop the innermost open tag; tags nest and inherit.

The defining rule is that **a bracket group that isn't a valid tag is emitted as literal text** — `"[INFO] items[0] [a-z]+"` renders as itself instead of raising (Rich's `MarkupError`) or silently disappearing. That's what makes `markup=True` defensible on a `Log`, whose lines usually come from somewhere other than the person who enabled it; `escape()`/`\[` forces a literal bracket that *would* have parsed.

`render(markup, base)` returns `(text, Style)` runs; `plain()`, `escape()`, `split_lines()`, `slice_runs()` (by *character* index) and `write_runs(canvas, x, y, runs)` are the shared helpers the four widgets are built from. Each widget caches its parse against a key that includes the base style's `(fg, bg, styles)` — not just the text — because runs bake the base in and a theme switch mutates `App.style` **in place** (see "Theming"), so an identity check would keep the old colors forever. `Text` builds its plain lines and its run-lines in one pass (`_build`), slicing runs by the plain segments' lengths; that's only sound because `textwrap.wrap(..., drop_whitespace=False, replace_whitespace=False)` loses no characters, so a paragraph's wrapped segments concatenate back to the paragraph exactly. `AnimatedLabel` merges per-character: the animation keeps the foreground only if it *changed* it from the base it was handed, so `GlowAnimation`/`RainbowAnimation` (which animate fg) still win on color while contributing nothing to bg/attrs, and `LevitateAnimation` (which passes the base through) lets the tag's color show.

`Log` (`widgets/display/log.py`) is a `ScrollView` subclass that owns its rows: `log(*values, sep=" ")` is print-shaped, splits embedded newlines into separate rows, and appends a `Label` per line. History is a `deque(maxlen=max_lines)`; on overflow the oldest child is popped and the rest renumbered, because child `y` *is* the row index that `ScrollView` measures content height and scroll offsets against. Deliberately distinct from `_devtools.py`'s `_LiveLog`, which rebuilds every row from `App._debug_log` on a seq change — fine for a rarely-open debug pane, far too much per-line work for a widget an app logs to continuously.

### Reactive state (`state.py`)
`State(value)` is an observable box: `.value`'s setter notifies subscribers (`set()`/`update(fn)` are the callable forms), skipping the notification when the new value compares equal. `Widget.bind(attr, value)` is the integration point — widgets call it in `__init__` in place of a bare assignment (`self.bind("text", text)`), and it does nothing but `setattr` for a plain value, so a non-`State` construction path is unchanged apart from one `isinstance` check. Given a `State` it delegates to `State.bind(obj, attr)`, which resolves to the current value *and* subscribes; **widgets always store plain values**, never the `State` — so every `draw()`, `natural_width()`, and existing widget that was never taught about states keeps working untouched. Assignment goes through `setattr`, so a property-backed target (`Text.text` re-wraps, `ProgressBar.progress` clamps) reacts exactly as it would to a hand-written assignment. Reactive properties today: `Label`/`Button`/`Checkbox.text`, `Hyperlink.text`/`link`, `Box.text`/`title`, `Text.text`, `ProgressBar.progress`.

Widget bindings hold the widget **weakly** (`State.subscribe(cb, owner=widget)`, pruned in `_notify`) — without that, a long-lived `State` in an app that continuously spawns and drops widgets (TermQuarium's fish) would pin every one of them plus its parent chain. Deliberately absent: no dependency tracking, no proxying, no computed/derived states, and no two-way binding (an `Input` writing back is `input.on_change(state.set)`, kept explicit). **No `App` plumbing exists or is needed**: notification is synchronous, and the repaint is the loop's ordinary next frame — which already follows key/mouse dispatch, timer callbacks, and worker `on_result`, i.e. everywhere user code runs — with the diff renderer emitting only changed cells. The one gap is setting a `State` straight from a worker thread: attributes update but no frame is scheduled, the same constraint that already governs touching widgets off the main thread.

### Animation (`motion.py`)
Easing curves (`linear`/`ease_in`/`ease_out`/`ease_in_out`/`ease_out_quad`/…), value + color interpolation, and a self-timing `Tween`. There is no animation *driver* — the loop is event-driven and idles at zero CPU, so an animating widget pulls frames itself: hold a `Tween`, and from `draw()` read `tween.value()` and call `canvas.request_frame(0.033)` until `tween.done`. `AnimatedLabel` and `DevToolsPanel` are the in-tree examples of that pattern.

### Background work
`app.run_worker(func, on_result=, on_error=)` runs `func` on a daemon thread; results are queued and the callbacks fire on the **main thread** from the event loop (never touch the UI from a worker thread directly).

### Clipboard (`clipboard.py`)
Built-in, no `pyperclip`. Picks a native backend per platform (Win32 API, `pbcopy`/`pbpaste`, `wl-clipboard`/`xclip`/`xsel`) with an OSC 52 fallback. `clipboard.backend()` reports the chosen one.

### Screen export (`_export.py`)
`App.export_svg`/`export_html` render the *current* screen (call `_compose()` first, as these do) into a standalone, colors-and-styles-baked-in SVG/HTML document — `snapshot()` with the visuals kept, for docs/READMEs/issue reports. `App.save_screenshot(path=None, format=None)` writes one to disk (format inferred from `path`'s extension, defaulting to an auto-timestamped `.svg` under `~/.cozy_tui/screenshots/`, created if missing) and is what **Ctrl+S** (`App._quick_screenshot`, bound unconditionally like Ctrl+T/Ctrl+P, not gated on `debug=True`) calls, reporting success/failure via a toast rather than raising out of the key handler. Colors resolve through `ansi.resolve_rgb()` — the same lookup the real terminal renderer uses to downgrade truecolor/256-color values — so every color form (named, `#rrggbb`, `rgb(R,G,B)`, `color(N)`) works with no separate export-only color table. Both renderers walk `App.buffer` emitting maximal same-style runs per row (mirroring the diff-renderer's own run-based thinking) rather than one element per cell; SVG runs use `textLength`/`lengthAdjust="spacingAndGlyphs"` to keep the monospace grid aligned even across wide glyphs.

`frame=` ("macos"/"windows"/"gnome", default `None`) wraps the raw grid in a fake OS window — titlebar (traffic-light dots / window buttons / a single close button, per `_FRAMES`), rounded corners, and a drop shadow (SVG: `feDropShadow` + a `clipPath` so square cell content still gets rounded corners; HTML: `box-shadow`/`border-radius`/`overflow:hidden` on a wrapping div) — so an export reads as a window screenshot instead of text floating in a corner. `export_html(standalone=True)` goes one step further: a dark, centered page around the (possibly framed) terminal, responsive via `overflow:auto` (scrolls rather than breaking layout on narrow viewports) — and gives even a frameless terminal its own soft rounded/shadowed box, so opening the file directly is presentable too. `App._quick_screenshot` (Ctrl+S) passes `frame="macos"` by default for exactly this reason.

### Code / CodeInput (`widgets/display/code.py`, `widgets/input/code_input.py`)
`Code(code, lang="python")` is a syntax-highlighted source block via Rich's `Syntax` (Pygments), bridged through `_rich_bridge` like `Markdown`/`TracebackView`. **Position is keyword-only** (`x=`/`y=`) so the source reads first — the one widget in the library that isn't `Widget(x, y, ...)`, so its constructor takes a `*_positional` catch-all purely to turn that likely mistake into a named `TypeError` instead of an arity error or a failure deep inside Pygments. With no `width` it self-sizes by rendering once at a generous probe width and reading back how much was actually used (`_measure()`) — the line-number gutter's exact width is Rich's business and has changed shape across versions, so measuring the rendered output is the one number that can't drift out of sync with it. Default theme is `"ansi_dark"`, not Rich's `"monokai"`: its colors *are* the 16 ANSI names, so they survive the bridge's nearest-color mapping exactly. An unknown `lang` renders unhighlighted rather than raising (a viewer must not crash mid-draw over a lexer name). Everything reads through `_sync()`, whose cache key covers every input, so mutating `code`/`lang`/`line_numbers` as plain attributes is enough to update it and an idle block costs nothing per frame. `code` accepts a `State`.

`CodeInput(Input, Code)` is the editable version, following `MarkdownInput(Input, Markdown)` exactly: focused → `Input.draw` (cursor/selection/scrolling intact), unfocused → `Code.draw`. It deliberately does **not** call `Code.__init__` (which would assign `code`, a read-only property here) and instead seeds Code's render state by hand, the same way `MarkdownInput` seeds its own cache fields; `code` is a read-only view onto `Input.value` so there's one source of truth and the cache key follows the text actually being edited. Note the naming split with `Diff`, which takes `lexer=` for the same concept that `Code`/`CodeInput` call `lang=`.

### Rich bridging (`_rich_bridge.py`)
Shared by every widget that borrows Rich's rendering instead of reimplementing it (`Markdown`, `TracebackView`, `Code`, `Diff`): `to_cozy_style(rich_style, base)` converts a resolved `rich.style.Style` into a `cozy_tui.style.Style`, mapping colors down to the 16 named ANSI colors (via `cozy_color()`) regardless of Rich's original color space — `ansi.py`'s own depth-aware downgrading takes it from there. Widgets render through a headless `rich.console.Console` (`console.render_lines(...)`), never printing Rich's own ANSI output to stdout, so it composes with the raw-mode/alt-screen renderer instead of corrupting it. `TracebackView` (`widgets/display/traceback_view.py`) applies this to `rich.traceback.Traceback`; `crash_screen.show_traceback(exc)` wraps it into a ready-made full-screen crash view (Esc quits, C copies via `clipboard.copy`).

## Examples (`examples/`)
Each example is a self-contained folder run directly (`python examples/<name>/<name>.py`, or `cozy-tui run …`) — they are the library's stress tests as much as its demos, and several have real test coverage in `tests/`. They import `cozy_tui` as an installed package; nothing in `cozy_tui/` imports back.

### TermQuarium (`examples/aquarium/`)
By far the largest example — a full aquarium-sim game, and the main place this library gets exercised at scale (thousands of lines, hundreds of live widgets, continuous `tick_interval` animation). Structure:

- `aquarium.py` — the runnable entry point and the whole UI/game-loop layer (screens, input, steering blends, spawning). Kept as the single entry file on purpose; its module docstring is a chronological "Step N / Phase N" design log explaining *why* each mechanic works the way it does — read it before changing behavior.
- `termquarium/` — the game systems factored out so `aquarium.py` doesn't have to hold everything: `constants.py` (all tunables + shop/species/treat tables), `fish.py`, `steering.py`, `relationships.py`, `economy.py`, `shop.py`, `world.py` (day/night + water temperature off one shared 0..1 day fraction), `tank_objects.py`, `bubbles.py`/`leaves.py`/`dreams.py`/`vignettes.py` (ambience), `inspectors.py`/`ui.py`/`styles.py` (panels), `save.py`, `cloud.py`, `console.py`.
- `save.py` — versioned plain-JSON saves under `~/.termquarium/saves/` (`SAVE_VERSION`, `safe_filename()`). Deliberately inspectable/shareable; migrate rather than break the format.
- `console.py` — the backtick **Cheat Console** for setting up a scenario in seconds during testing. `parse_command()` is a *structural* `ast.parse(mode="eval")` + per-argument `ast.literal_eval()`, never `eval`/`exec` — a typed command can only ever pass plain literals. Keep it that way when adding commands.
- `cloud.py` + `website/api/` — optional cloud saves. The client is stdlib `urllib` on `app.run_worker()` threads (no asyncio, no new dependency); the server is a small FastAPI app deployed as a Vercel Python function (`index.py` routing + auth, `store.py` logic). Auth is a locally-generated **Cloud Key** sent as `Authorization: Bearer …` and used directly as the storage namespace — no accounts, no login. `_safe_name()` in `index.py` intentionally duplicates `save.py`'s sanitization (separate deployment, can't import it).
- `website/` — the static marketing/docs site (plus `documentation/`), deployed separately via the `vercel.json` files.

Tests: `tests/test_aquarium.py` (steering/movement), `test_termquarium_world.py`, `test_termquarium_save.py`, `test_termquarium_console.py`, `test_termquarium_cloud.py`, `test_termquarium_cloud_api.py`.
